# Check Fix

对当前修改做一次完整的修复影响分析，重点找回归风险、兼容性问题、遗漏测试和潜在副作用。

## 工作流

1. 先确认变更范围。
- 优先读取当前 `git diff`
- 如果没有 diff，就根据当前对话和最近改动文件推断范围

2. 分析直接影响。
- 哪些函数、方法、配置项、接口、路由被修改
- 哪些调用方会直接受到影响
- 参数签名、返回值、错误处理、日志行为是否变化

3. 分析间接影响。
- 调用链上下游是否受影响
- 共享状态、缓存、并发、回调、事件时序是否变化
- 是否引入新的性能、权限、资源释放问题

4. 分析兼容性。
- 新增字段是否有默认值
- 删除字段或改类型后，旧代码、旧数据、下游调用方是否还兼容
- 配置项、序列化格式、环境变量是否出现不兼容变更

5. 盘点验证覆盖。
- 现有测试覆盖了什么
- 缺少哪些关键测试
- 是否需要补手工验证或联调验证

6. 给出结论。
- 风险等级
- 最需要补的验证项
- 是否建议继续合并、发布或继续修改

## 输出格式

```markdown
📝 变更摘要
- 修改文件:
- 修改类型:
- 影响范围:
- 风险等级:

🎯 直接影响
- ...

🔗 间接影响
- ...

🧱 数据与兼容性
- ...

🧪 测试与验证
- Existing:
- Missing:

⚠️ 风险点
- ...

✅ 建议动作
- ...
```

## 约束

- 不确定的地方要明确说明不确定性来源。
- 没有证据不要断言“没有影响”。
- 如果改动很小，也要说明为什么风险低。
